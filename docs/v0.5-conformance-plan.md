# MTP v0.5 Plan

Implemented in `v0.5.0`. This document is retained as the design record for the conformance release.

## Why v0.5 Needs a Scope Correction

The current roadmap still describes `v0.5` as "drift scoring engine, conformance test suite".
That is no longer accurate.

As of the current `v0.4.x` state, the repository already contains:

- weighted self-drift scoring in `mtp-run score`
- weighted cross-report drift comparison in `mtp-run drift`
- adapter-aware execution runtime in `mtp-run exec`
- fixture-backed unit tests for `mtp-run` and `mtp-lint`
- CI coverage for both tooling packages

Therefore the real missing `v0.5` deliverable is not "build a drift engine".
It is:

**turn the existing tools into a formal, fixture-driven MTP conformance suite.**

## What v0.5 Should Deliver

### 1. Conformance Fixtures

Add a first-class fixture tree, for example:

```text
conformance/
  fixtures/
    l1-packages/
      valid/
      invalid/
    l2-execution/
      success/
      partial/
      deviation/
      failure/
      skipped/
      escalated/
    l3-redaction/
      pii/
      secrets/
      client-identifiers/
      regulated/
    drift/
      self-score/
      cross-report/
```

Each fixture should include:

- input artifact(s)
- expected result
- optional notes explaining why the fixture exists

### 2. Conformance Runner

Introduce a dedicated runner, preferably a new tool:

```text
tools/mtp-conformance/
```

Why a separate tool:

- `mtp-lint` owns validation and redaction
- `mtp-run` owns execution and drift
- conformance must orchestrate both without overloading either CLI

Suggested commands:

```bash
mtp-conformance run --level l1
mtp-conformance run --level l2
mtp-conformance run --level l3
mtp-conformance run --format json
```

### 3. Formal Level Evaluation

Codify the spec's conformance levels as executable rules:

- `L1`: package schema validity on valid fixtures, rejection of invalid fixtures
- `L2`: correct execution-report state handling across all six execution states
- `L3`: L1 + L2 + redaction detection + drift scoring + provenance presence

The runner output should include:

- level
- total fixtures
- passed
- failed
- per-fixture results
- summary hash

### 4. Golden Execution Scenarios

Current tests are unit tests. `v0.5` needs scenario fixtures.

At minimum, add packages or execution reports that deterministically exercise:

- `success`
- `partial`
- `deviation`
- `failure`
- `skipped`
- `escalated`

The mock adapter already supports marker-driven outcomes, so it can generate most of these fixtures reliably.

### 5. Redaction Conformance Pack

`mtp-lint` already has detection logic. `v0.5` should formalize it as a conformance corpus:

- package with obvious email leak
- package with secret/token leak
- package with client dictionary hit
- package with regulated-content hit
- package that is clean and must pass

### 6. Drift Conformance Pack

Because drift scoring now exists, `v0.5` should freeze reference cases.

At minimum:

- perfect mock report -> expected composite `1.0`
- one-deviation report -> expected self-score `0.92`
- mock vs example report -> expected cross-report drift `0.9111`

These values should be asserted by the conformance runner, not left only in unit tests.

### 7. CI Promotion

Extend CI so conformance becomes a release gate:

- unit tests remain fast correctness checks
- conformance runner becomes release-level verification

Suggested CI jobs:

- `conformance-l1`
- `conformance-l2`
- `conformance-l3`

## Recommended Implementation Order

### Phase 1

- correct roadmap language
- add `docs/v0.5-conformance-plan.md`
- create fixture directory layout
- create first fixture manifest format

### Phase 2

- build `mtp-conformance` runner
- implement L1 fixture execution
- implement JSON output and exit codes

### Phase 3

- add L2 execution fixtures using mock adapter markers
- add drift reference fixtures
- make CI run the conformance tool

### Phase 4

- add L3 redaction fixtures
- produce machine-readable release summary
- document conformance claims in README

## Acceptance Criteria for v0.5

`v0.5` should not be considered done until all of the following are true:

- a dedicated conformance fixture tree exists in-repo
- conformance can be run from one CLI entrypoint
- L1/L2/L3 have executable pass/fail rules
- drift reference cases are asserted against exact expected scores
- CI runs the conformance suite automatically
- README reflects conformance-first `v0.5`, not "future drift engine"

## Non-Goals for v0.5

These should stay out of scope:

- registry/signatures/approvals -> `v0.6`
- benchmark publication across external platforms -> `v1.0`
- mandatory live-cloud adapter tests in CI

Real-adapter smoke tests are useful, but they should stay optional because they depend on secrets and external APIs.
